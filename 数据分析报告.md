# 碳足迹计算系统数据分析报告

## 1. 数据模型定义分析

### 1.1 核心数据表结构

通过分析 `backend/schema.sql` 文件，系统定义了三个核心数据表：

#### Factors 表（因子表）
```sql
CREATE TABLE IF NOT EXISTS factors (
  id VARCHAR(64) PRIMARY KEY,           -- 因子唯一标识符
  category VARCHAR(64),                 -- 类别（transport, electricity, food等）
  unit VARCHAR(32),                     -- 单位（km, kWh, kg等）
  value DOUBLE,                         -- 因子数值（gCO2e/单位）
  source VARCHAR(256),                  -- 数据来源
  region VARCHAR(64),                   -- 地区标识
  version VARCHAR(32),                  -- 版本号
  effective_date DATE,                  -- 生效日期
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### Ledger 表（账本表）
```sql
CREATE TABLE IF NOT EXISTS ledger (
  id BIGINT PRIMARY KEY AUTO_INCREMENT, -- 主键
  user_id VARCHAR(128),                 -- 用户标识
  type VARCHAR(64),                     -- 活动类型
  amount DOUBLE,                        -- 数量
  unit VARCHAR(32),                     -- 单位
  meta JSON,                           -- 元数据（JSON格式）
  kgco2e DOUBLE,                       -- 计算结果（kg CO2e）
  scope VARCHAR(32),                   -- 范围分类
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### Badges 表（徽章表）
```sql
CREATE TABLE IF NOT EXISTS badges (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id VARCHAR(128),
  badge_id VARCHAR(128),
  metadata JSON,
  awarded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY user_badge_unique (user_id, badge_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 1.2 数据模型特点
- **标准化设计**：采用统一的数据结构存储不同类型活动
- **扩展性强**：使用JSON字段存储元数据，便于扩展
- **版本控制**：因子表包含版本信息，支持因子更新追踪
- **地域支持**：内置地区字段，支持多地区数据区分

**优点：**
- 包含版本字段（version）
- 有生效日期字段（effective_date）

**不足：**
- 缺少因子变更历史追踪机制

## 4. 计算引擎分析

### 4.1 核心计算逻辑
计算引擎位于 `shared/src/calculator.js`，实现了基础的碳足迹计算：

```javascript
// 运输计算
function transportKgCO2e(km, gco2PerKm) {
  return gramsToKg(km * gco2PerKm);
}

// 电力计算
function electricityKgCO2e(kwh, gco2PerKwh) {
  return gramsToKg(kwh * gco2PerKwh);
}

// 食物计算
function foodKgCO2e(kgFood, gco2PerKg) {
  return gramsToKg(kgFood * gco2PerKg);
}

// 通用活动计算
function calculateActivity(activity) {
  switch (activity.type) {
    case 'transport': return transportKgCO2e(activity.amount, activity.meta.gco2PerKm);
    case 'electricity': return electricityKgCO2e(activity.amount, activity.meta.gco2PerKwh);
    case 'food': return foodKgCO2e(activity.amount, activity.meta.gco2PerKg);
    case 'purchase': return purchaseKgCO2e(activity.amount, activity.meta.gco2PerCurrencyUnit);
    default: return 0;
  }
}
```

### 4.2 计算准确性评估
**优点：**
- 使用共享库确保前后端计算一致性
- 支持多种活动类型的计算
- 单位转换处理正确（克转千克）

**不足：**
- 计算公式相对简单，缺少复杂场景支持
- 缺少计算过程的日志记录
- 没有计算结果的验证机制
- 缺少不确定性范围表示

### 4.3 验证机制缺失
目前系统缺乏：
- 计算结果的交叉验证
- 异常值检测机制
- 计算过程的审计跟踪

## 6. 综合评估与建议

### 6.1 优势总结
1. **架构清晰**：前后端分离，共享计算逻辑
2. **数据结构合理**：标准化表设计，易于扩展
3. **基础功能完备**：支持主要碳足迹计算场景
4. **国际化起步**：已有多语言框架

### 6.2 主要问题
1. **因子管理薄弱**：缺少完善的版本控制和变更追踪
2. **数据治理不足**：缺乏数据质量保证机制
3. **计算能力有限**：算法简单，缺少高级特性
4. **用户体验待优化**：单位转换、地区适配需要完善

### 6.3 优先级改进建议

**高优先级：**
- 实现因子版本管理增强
- 添加数据导入验证机制
- 完善计算结果验证

**中优先级：**
- 扩展单位转换支持
- 增强多地区适配能力
- 优化国际化体验

**低优先级：**
- 添加高级计算算法
- 实现复杂的因子依赖管理
- 开发数据可视化分析功能

## 7. 结论

该碳足迹计算系统具备良好的基础架构，在数据模型设计和核心功能实现方面表现稳定。但在因子管理、数据质量和用户体验等方面仍有较大提升空间。建议按照上述优先级逐步完善系统功能。
